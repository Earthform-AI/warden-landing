## 2. /beacon Page — Tier 1: Dashboard

### What exists today

Three production API endpoints on earthform.ai:

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/api/beacons/emit` | POST | Create a PnR routing beacon (device config + design + seed count) |
| `/api/beacons/pending` | GET | Workers poll for next seed to route |
| `/api/beacons/fulfill` | POST | Record PnR result (fmax, timing, util) with hash-chained ledger |

Supabase tables: `routing_beacons`, `routing_results`

### Dashboard page: `/beacon`

A read-only page that shows the live state of the routing beacon system.

**Sections:**

#### 2A. Hero
- Title: "Beacon Network"
- Subtitle: "Distributed place-and-route for open hardware. Watch real designs converge."
- Stat bar: total beacons emitted, seeds completed, best fmax achieved

#### 2B. Active Campaigns
- Fetch from `routing_beacons` where status in ('pending', 'active')
- Per beacon card:
  - Device target (e.g., "ECP5-85F CABGA381")
  - Frequency target (e.g., "25 MHz")
  - Progress bar: seeds_completed / seeds_requested
  - Best fmax so far (green if timing_met, amber if close, red if far)
  - Channel badge

#### 2C. Recent Results Feed
- Fetch latest N from `routing_results` ordered by created_at desc
- Per result row:
  - Seed value, fmax achieved, timing_met badge, route time
  - Contributor label (anonymous if none)
  - Hash chain indicator (shows this result chains to previous)

#### 2D. How to Contribute
- Brief explainer: what PnR is, why seeds matter
- Prerequisites: nextpnr-ecp5 installed, ~2 min per seed
- Link to a CLI tool or script that polls `/api/beacons/pending` and runs nextpnr
- "More about the architecture" → link to /transparency

#### 2E. Network Stats (bottom)
- Total seeds routed all-time
- Average fmax across all completed beacons  
- Unique contributors
- Hash chain depth (total ledger entries)

**Implementation approach:**
- Astro page with `fetch()` to beacon API at build time (SSR) or client-side
- Prefer client-side fetch for live data (the page should feel alive)
- React component for the results feed (auto-refresh every 30s)
- Tailwind styling consistent with rest of site (dark theme, blue/green accents)

**New files:**
- `src/pages/beacon.astro` — Page shell + SSR initial data
- `src/components/BeaconDashboard.tsx` — React component for live data
- No config file needed — data comes from API

**Nav update:** Add "Beacon" to nav in `site.config.ts` (between Learn and About, or after Learn)

## 3. /beacon — Tier 2: Fano-7 Seed Recommender

### Concept

The Fano-7 model (205K params, trained on convergence prediction) becomes an active participant in the beacon system. Instead of golden-ratio seed spread (blind), the model predicts which seeds are likely to meet timing on a given board.

### D174 Experiment (CAExperiments)

**Note:** D174 was originally planned for tone quadrant. Use **D177** or the next available number for the seed recommender experiment if D174 is taken.

**Hypothesis:** A Fano-7 model fine-tuned on historical `routing_results` data (seed → fmax outcome per device/design pair) predicts fmax within ±2 MHz and recommends seeds that achieve timing_met at >80% rate (vs ~50% baseline with golden-ratio spread).

**Inputs:**
- `routing_results` table export: seed, fmax_mhz, timing_met, device_size, freq_target, lut_util, route_time
- Fano-7 base model weights from `fano7-hf/pytorch_model.bin`

**Architecture:**
- Fano-7 attention (7 heads on PG(2,2)) with associator gating
- Input: seed features (7-element vector: seed value decomposed into Fano-plane components) + device/design features
- Output: predicted fmax, confidence, recommended next seeds
- Training: fine-tune on routing_results, evaluate on held-out seeds

**Integration with beacon system:**
- New endpoint: `GET /api/beacons/recommend?beacon_id=X` returns model-recommended seeds
- `pending.ts` gains optional `strategy=model` parameter (default: `golden`)
- Model runs as serverless function or edge function (205K params = tiny, fits in Vercel)
- Crystallization: as more results confirm predictions, model confidence increases → fewer seeds needed per beacon

**Success criteria:**
- Seeds recommended by model achieve timing_met at >80% vs ~50% baseline
- Average seeds-to-timing reduced from 8 to ≤4
- Model inference time < 100ms per recommendation

### Implementation order
1. Export routing_results data (need sufficient volume — at least 50+ results)
2. Feature engineering: seed → 7-element Fano decomposition
3. Fine-tune Fano-7 on routing outcomes
4. Evaluate on held-out seeds
5. Deploy as `/api/beacons/recommend` endpoint
6. Update dashboard to show model confidence alongside golden seeds
